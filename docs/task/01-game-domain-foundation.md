# 01 Game Domain Foundation 設計書

## 目的

グリッド上にマシンを配置し、マシン同士を方向付き接続で結ぶためのゲーム状態を定義します。以降のレシピ、シミュレーション、操作、描画はこの状態を参照します。

## 変更予定のフォルダ構成

```text
apps/web/src/game
├── grid.ts
├── grid.test.ts
├── placement.ts
├── placement.test.ts
├── connections.ts
├── connections.test.ts
├── gameState.ts
└── gameState.test.ts
```

## ファイルごとの責務

- `grid.ts`
  - `(x, z)` の整数グリッド座標型を定義します。
  - grid key 生成、同一座標判定、ワールド座標変換、隣接方向計算を提供します。
- `placement.ts`
  - 配置物 ID、配置物種別、配置済みマシンの型を定義します。
  - 重複配置の禁止、配置、削除、選択対象検索を提供します。
- `connections.ts`
  - マシン間の方向付き接続を定義します。
  - 接続作成、削除、入力元・出力先検索、重複接続防止を提供します。
- `gameState.ts`
  - 配置物、接続、ステージ進行、選択状態をまとめる `GameState` を定義します。
  - 初期状態生成と state 更新関数を提供します。

## 実装のポイント

- グリッド座標は `{ x: number; z: number }` とし、整数のみを有効値として扱います。
- `gridKey` は `x:z` 形式の文字列に統一し、Map の key とテスト期待値を安定させます。
- マシン自体は向きを持たせません。素材の流れは `connections.ts` の接続方向で表現します。
- 同一セルに配置できる配置物は 1 つのみです。
- 接続は `fromMachineId` と `toMachineId` を持つ有向辺です。
- 削除時は対象マシンに接続する入力・出力接続も同時に削除します。
- UI や Three.js に依存しない純粋関数として実装します。

## パッケージ利用範囲

- 使用する既存パッケージは Vitest のみです。
- グリッド、配置、接続のルールはゲーム固有のため自作します。

## テスト観点

- グリッド座標を key 化し、同じ座標が同じ key になること。
- 非整数座標を配置入力として扱わないこと。
- 空セルへ配置でき、既存配置セルへ重複配置できないこと。
- 配置削除で対象マシンと関連接続が消えること。
- 同じ接続を二重作成できないこと。
- 存在しないマシンを接続元または接続先にできないこと。
